## 1. Purpose & scope

Stand up the WATCHMOGGED monorepo so that Week 1 (schema + auth) can begin without re-litigating tooling, packaging, or deployment plumbing.

**In scope:**
- Turborepo + pnpm workspaces root
- `apps/web` — Next.js 15 (App Router) + TypeScript strict + Tailwind v4 + shadcn/ui
- `apps/mobile` — Expo (current stable SDK at scaffold day) + TypeScript strict + NativeWind + Expo Router
- `packages/{api,db,types,utils,config}` — stubs with shared TS configs
- `supabase/` — local CLI init only
- GitHub Actions CI — typecheck + lint + test on PR and push to `main`
- Vercel hello-world deployment for web (preview on PR; production on `main`)
- Expo Go hello-world for mobile (no EAS in Week 0)
- Vitest workspace harness (one real test in `packages/utils`)
- Cross-package smoke import exercised by both apps
- Surgical CLAUDE.md edit to reflect the feature-branch PR model
- Branch protection rules on `main`

**Explicitly out of scope** (deferred to later weeks):
- Any product schema, RLS policy, or Edge Function
- Creating staging or production Supabase projects (Week 1)
- EAS Build / EAS Update / EAS Submit (first natural fit: Week 6)
- Stripe / Resend / Sentry / PostHog / Anthropic SDK integration (accounts tracked in SETUP.md parallel work)
- Husky, lint-staged, commitlint
- Turbo Remote Caching
- Any UI beyond the smoke page
- Custom domain pointed at Vercel
- Mobile Vitest setup (RN testing belongs with the first non-RN-module test, not now)

### 3.1 Tailwind / NativeWind compatibility note

NativeWind has historically tracked Tailwind v3 syntax internally even after Tailwind v4 shipped. At scaffold day we use whatever Tailwind version NativeWind's current stable requires. **Web and mobile do not share a Tailwind config**, so a version mismatch between the two is intentional and not a bug. Flagging only to prevent future confusion.

### 3.2 `tsconfig.base.json`

```jsonc
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "lib": ["ES2022"],
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "noImplicitOverride": true,
    "noFallthroughCasesInSwitch": true,
    "isolatedModules": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "resolveJsonModule": true,
    "verbatimModuleSyntax": true
  }
}
```

Apps override `lib`/`jsx`/`types` as needed (web adds DOM and Next.js types; mobile adds React Native types). `verbatimModuleSyntax: true` is intentional — strict type-import discipline keeps Supabase-generated types out of runtime bundles and pays for itself given two bundlers consume `packages/api`.

### 3.3 ESLint flat config

Single root `eslint.config.mjs`:
- Base: `@eslint/js` recommended + `typescript-eslint` recommended-type-checked
- Plugins: `eslint-plugin-import` (ordering), selective `eslint-plugin-unicorn` rules (only those that do not fight Prettier)
- Apps layer their own:
  - Web extends with `next/core-web-vitals`
  - Mobile extends with the Expo lint config
- No per-package `.eslintrc` files

### 3.4 Prettier

Root `.prettierrc` with defaults + `prettier-plugin-tailwindcss` for class sorting. Format-on-save handled by editor config; Husky/lint-staged deferred.

### 3.5 Vitest workspace

Root `vitest.workspace.ts` discovers `packages/*/vitest.config.ts` and (eventually) `apps/web/vitest.config.ts`. Mobile is excluded — adding Vitest to a React Native project is its own setup (`jest-expo`) and not worth solving in Week 0 with zero tests.

Per-package `test` script: `vitest run --passWithNoTests`. This keeps CI green for packages with no tests yet.

### 3.6 Turbo pipeline (`turbo.json`)

```jsonc
{
  "$schema": "https://turbo.build/schema.json",
  "tasks": {
    "lint":      { "dependsOn": ["^lint"] },
    "typecheck": { "dependsOn": ["^typecheck"] },
    "test":      { "dependsOn": ["^test"] },
    "build":     { "dependsOn": ["^build"], "outputs": [".next/**", "dist/**"] },
    "dev":       { "cache": false, "persistent": true }
  }
}
```

Remote caching is **not** configured in Week 0. Defer.

Root `package.json` declares the passthrough scripts:

```jsonc
{
  "scripts": {
    "lint": "turbo run lint",
    "typecheck": "turbo run typecheck",
    "test": "turbo run test",
    "build": "turbo run build",
    "dev": "turbo run dev"
  }
}
```

Anywhere this spec refers to `pnpm <task>` at the repo root, it means this passthrough.

## 4. Package definitions

Each package follows the same shape: `package.json` with `main`/`types` pointing at TS source, `tsconfig.json` extending `tsconfig.base.json`, and a `src/index.ts`.

### 4.1 Canonical `package.json` shape

```jsonc
{
  "name": "@watchmogged/<name>",
  "version": "0.0.0",
  "private": true,
  "type": "module",
  "main": "./src/index.ts",
  "types": "./src/index.ts",
  "exports": {
    ".": "./src/index.ts"
  },
  "scripts": {
    "lint": "eslint .",
    "typecheck": "tsc --noEmit",
    "test": "vitest run --passWithNoTests"
  }
}
```

### 4.2 Stub contents per package

| Package | `src/index.ts` contents |
|---|---|
| `@watchmogged/utils` | Real `formatUsd(amountCents: number): string` |
| `@watchmogged/types` | `export type Brand<K, T> = K & { readonly __brand: T };` |
| `@watchmogged/db` | `export type Database = { /* generated by supabase gen types in Week 1 */ };` |
| `@watchmogged/api` | `export {};` |
| `@watchmogged/config` | `export const APP_NAME = 'watchmogged' as const;` |

### 4.3 `formatUsd` implementation (the only real Week 0 code)

```ts
// packages/utils/src/index.ts
export function formatUsd(amountCents: number): string {
  return new Intl.NumberFormat('en-US', {
    style: 'currency',
    currency: 'USD',
  }).format(amountCents / 100);
}
```

Chosen because: trivial, deterministic across web and React Native runtimes (Intl is in both), and exactly the kind of helper Week 3+ will hit constantly. Cents-as-integer is the project's money representation convention from this point forward.

### 4.4 Vitest proof-of-life

```ts
// packages/utils/src/index.test.ts
import { describe, expect, it } from 'vitest';
import { formatUsd } from './index';

describe('formatUsd', () => {
  it('formats cents as USD with two decimals', () => {
    expect(formatUsd(12_500_000)).toBe('$125,000.00');
  });
  it('handles zero', () => {
    expect(formatUsd(0)).toBe('$0.00');
  });
});
```

This is the only test in Week 0.

## 5. App scaffolds

### 5.1 Web (`apps/web`)

- Scaffolded via:
  ```
  pnpm create next-app@latest apps/web \
    --typescript --app --tailwind --eslint --src-dir \
    --import-alias '@/*' --use-pnpm
  ```
  Then pruned to match our flat ESLint config and root Prettier.
- `next.config.ts` adds:
  ```ts
  transpilePackages: [
    '@watchmogged/api',
    '@watchmogged/config',
    '@watchmogged/db',
    '@watchmogged/types',
    '@watchmogged/utils',
  ]
  ```
- shadcn/ui initialized: `pnpm dlx shadcn@latest init`
  - Style: New York
  - Base color: neutral
  - CSS variables: yes
  - Components dir: `apps/web/src/components/ui/`
- Single shadcn `Button` installed as a registry smoke check.
- Tailwind v4: `apps/web/src/app/globals.css` uses `@import "tailwindcss"` plus a minimal `@theme` block.
- Page lives at `apps/web/src/app/page.tsx` (the `--src-dir` flag nests `app/` under `src/`).

### 5.2 Mobile (`apps/mobile`)

- Scaffolded via:
  ```
  npx create-expo-app@latest apps/mobile -t default
  ```
  Whatever the current stable SDK is at scaffold day.
- NativeWind installed and wired per its current docs (Babel plugin, `tailwind.config.js`, `global.css` imported by the root layout).
- `metro.config.js` extended per Expo's monorepo guide:
  - `watchFolders` includes the workspace root
  - `nodeModulesPaths` resolves from both `apps/mobile/node_modules` and root `node_modules`
  - `disableHierarchicalLookup: true`
- Expo Router root layout + a single `app/index.tsx` screen.

## 6. CI workflow

`.github/workflows/ci.yml`:

```yaml
name: ci
on:
  pull_request:
    branches: [main]
  push:
    branches: [main]
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true
jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm typecheck
      - run: pnpm lint
      - run: pnpm test
```

- Single Linux job (no macOS/Windows matrix)
- Root scripts (`pnpm typecheck`, `pnpm lint`, `pnpm test`) are Turbo passthroughs that fan out per workspace
- No build step in CI — Vercel PR previews are the build gate

## 9. Branch model & branch protection

### 9.1 Model

- `main` is the integration branch.
- All work goes through short-lived feature branches → PR → squash-merge to `main`.
- All work — including solo work — goes through PRs.
- CI must pass to merge.
- No `dev` or `staging` branches.

Week 0 itself goes on `chore/week-0-monorepo-scaffold` and is PR'd into `main`.

### 9.2 Branch protection (manual GitHub UI step by the user)

Settings → Branches → Add rule for `main`:
- Require a pull request before merging
- Require status checks to pass — select the `verify` job from the `ci` workflow
- Require branches to be up to date before merging
- **Do not allow bypassing** (even for admins) — the discipline of the rule outweighs the convenience of bypass

Verification: attempting a direct push to `main` after rules are set must be rejected.

## 12. Risks and known unknowns

| Risk | Mitigation |
|---|---|
| NativeWind's current stable does not support Tailwind v4 | Web and mobile have independent Tailwind configs by design; mobile uses whatever NativeWind requires. No shared config exists to break. |
| Expo SDK has a regression at scaffold day | Spec does not pin SDK; we use whatever `create-expo-app@latest` produces. If hello-world fails, fall back to the previous SDK and document. |
| Vercel monorepo detection fails to install pnpm correctly | Manual override in Vercel UI: set Install Command to `pnpm install --frozen-lockfile` and Build Command to `pnpm turbo run build --filter=web`. |
| Docker Desktop not installed locally | Plan checks first and surfaces clear install instructions before running `supabase init` (init itself doesn't need Docker, but the user will need it for Week 1). |
| GitHub Actions billing on private repo | Free tier covers more than enough for this CI's footprint; flag if usage ever approaches the limit. |
